# 技术选型说明

## 为什么用 React（+ Vite + TypeScript）

- **组件化**适合搭建「表单式」上传与参数收集界面，而非画布编辑器。
- **Vite** 冷启动快，初学者改 UI 反馈迅速。
- **TypeScript** 能在前后端契约（模板结构、任务 ID）上提前发现错误。

## 为什么用 FastAPI

- Python 生态与**调用 FFmpeg 子进程**、文件 IO 自然契合。
- 自动生成 OpenAPI 文档，便于以后扩展字段与联调。
- 异步上传在后续大文件场景可逐步加强（Round 0 保持简单）。

## 为什么服务端调用 FFmpeg

- 合成、重编码、叠加 filter 在服务端一次完成，**终端一致、可复现**。
- 避免把重计算与编解码压到用户浏览器，降低兼容性与性能问题。
- 团队只需在服务器保证 FFmpeg 与字体依赖，而非每位用户环境。

## 为什么不优先使用 ffmpeg.wasm

- `ffmpeg.wasm` 适合特定「纯前端、小文件」场景，但 bundle 体积大、内存与多线程受限。
- 本项目明确以**服务端模板流水线**为主，与 wasm 路线不一致。
- 避免 Round 0 被浏览器编解码细节拖住。

## 为什么不做复杂编辑器

- 产品边界是**固定模板 + 自动合成**，复杂时间轴会指数级增加交互与状态管理成本。
- 初学者维护简单表单 + 后端任务，比维护非线性编辑器现实得多。

## 结构微调说明（相对用户初稿）

- 后端将路由拆到 `app/api/` 多个模块，便于按「健康检查 / 模板 / 任务」阅读；行为与初稿一致。
- 前端将页面放在 `pages/`，避免 `App.tsx` 臃肿；Round 0 仍保持单页。
